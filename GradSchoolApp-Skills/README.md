# GradSchoolApp-Skills

A collection of [Claude](https://claude.ai) skills for navigating PhD and MS graduate school applications, originally built for physics / optics / photonics / quantum applications but adaptable to any quantitative science field.

## What's included

| Skill | Purpose |
|---|---|
| **grill-me** | Structured interviewing that walks down the decision tree of a complex multi-part plan, one question at a time, with a recommended answer attached to every question. Field-agnostic. |
| **profile-builder** | Reads the user's CV, then grills them on the forward-looking information a CV doesn't carry, and produces a `profile.md` that the rest of the skill set depends on. |
| **program-discovery** | First-pass triage from the full universe of graduate programs down to a ranked working list of ~20–30 candidates. Writes `programs.md`. |
| **program-evaluation** | Deep evaluation of a single program against six weighted criteria, producing a rich scorecard and updating `programs.md`. Creates `essays/` and `outreach/` subfolders per program. |
| **sop-coach** | Develops, refines, and critiques Statement of Purpose drafts through probing questions and a six-check critique. Never writes SoP prose for the user. Contributes bio prose to `profile.md`'s narrative section. |
| **pi-cold-email** | Drafts and critiques outreach emails to potential PhD advisors across three modes (initial, reply, follow-up). Enforces required inputs, runs an eight-check critique, bakes in physics-specific conventions, and maintains `pi-contacts.md`. |
| **grad-school-app** | The orchestrator. Reads every artifact, computes application state, recommends what to work on this week, and regenerates `application-status.md` plus `calendar.ics` at root for calendar app import. |

The skills compose into a workflow:

```
   profile-builder    →  profile.md (applicant knowledge base)
         ↓
   program-discovery  →  programs.md (target program dashboard)
         ↓
   program-evaluation →  programs/{slug}/scorecard.md per program
                         + empty essays/ and outreach/ subfolders
                         + updates row in programs.md
         ↓                                    ↓
   sop-coach   →  core-sop.md           pi-cold-email   →  programs/{slug}/outreach/*.md
                  + sop-tailoring.md                       + pi-contacts.md
                  + bio prose into                           (cross-program log)
                    profile.md
         ↓                                    ↓
                      grad-school-app
                      reads everything above
                      writes application-status.md + calendar.ics
                      regenerated each invocation
```

`grill-me` is the underlying interview pattern used internally by several skills and invokable directly any time you face a complex multi-decision plan.

## Living documents and their owners

The skill set keeps several long-lived files at the project root, each with a single owner:

- **`profile.md`** (about the applicant) — owned by profile-builder; narrative section appended by sop-coach.
- **`programs.md`** (about the targets) — written by program-discovery, row-updated by program-evaluation.
- **`pi-contacts.md`** (about outreach state) — owned exclusively by pi-cold-email.
- **`core-sop.md`** (the core SoP draft) — owned by the user; sop-coach critiques only.
- **`application-status.md`** + **`calendar.ics`** — regenerated by grad-school-app every invocation.

Ownership boundaries prevent skills from clobbering each other's work.

## Write protocols

- **Narrative section in `profile.md`** is **append-only** — every session adds a timestamped block.
- **Factual sections in `profile.md`** (research, recommenders) use **clean rewrites with audit comments** (`<!-- last updated: YYYY-MM-DD by skill-name -->`).
- **`programs.md`** is row-scoped — program-evaluation only edits the row for the program being evaluated.
- **`pi-contacts.md`** is owned by pi-cold-email.
- **Scorecards** are clean rewrites per evaluation session.
- **Outreach drafts** are immutable per send (`{pi-slug}-{type}-{date}.md` is a record of what was drafted on that date).
- **`application-status.md` and `calendar.ics`** are full rewrites each grad-school-app invocation. Calendar uses stable UIDs so re-imports update events in place.

## How to use these skills with Claude

### Option 1 — Claude.ai Projects (easiest)

1. Create a new Project at [claude.ai](https://claude.ai).
2. Upload your CV.
3. Upload the seven `SKILL.md` files as project files. (Rename on upload if Claude's UI requires unique filenames, e.g. `sop-coach-SKILL.md`.)
4. In any conversation in the project, invoke skills naturally:
   - *"Build my profile"* → **profile-builder**
   - *"Find me PhD programs"* → **program-discovery**
   - *"Evaluate [program]"* → **program-evaluation**
   - *"Help me develop my SoP"* / *"Critique this draft"* → **sop-coach**
   - *"Draft a cold email to [PI]"* / *"Should I follow up?"* → **pi-cold-email**
   - *"What should I work on this week"* / *"Status update"* / *"Regenerate my calendar"* → **grad-school-app**
   - *"Grill me about [X]"* → **grill-me**

### Option 2 — Claude Code / API with skill auto-discovery and filesystem access

Place `GradSchoolApp-Skills/` somewhere Claude can read and give Claude write access to your project directory. The skills will then maintain all root files and per-program folders directly on disk.

### Option 3 — Manual prompting

Open a `SKILL.md`, paste its contents at the start of a conversation, and ask Claude to follow it. Works in any Claude interface but loses file-persistence.

## Suggested workflow

1. **Drop your CV in.**
2. **profile-builder** → `profile.md`.
3. **program-discovery** → `programs.md`.
4. **Iterate the program list** — drop misfits, add missed.
5. **program-evaluation** on each finalist (one conversation per program) → scorecards + folder structure.
6. **pi-cold-email** for summer outreach (July–October).
7. **sop-coach** — cluster grilling populates narrative; you draft `core-sop.md`; skill critiques. Tailoring blocks per program follow.
8. **grad-school-app** — run weekly to regenerate the status dashboard and `calendar.ics`. Import the calendar file into your preferred calendar app.
9. **grill-me anytime** — recommender choices, application order, admit comparisons.

## Adapting to other fields

- **`program-discovery` / `program-evaluation`**: swap "Optica rankings" for your field's specialty rankings; swap community sources.
- **`program-evaluation`**: adjust criterion #2 ("Program quality in optics-photonics") to your field's specialty metric.
- **`sop-coach`**: cluster #7 (cross-pollination from history-historiography) is user-specific; replace or drop.
- **`pi-cold-email`**: physics-specific conventions (season, per-program cap, no-attachment etiquette) may differ for other fields.
- **`profile-builder`, `grill-me`, `grad-school-app`**: field-agnostic.

## Anti-patterns these skills are designed to avoid

- Generic ranking advice that ignores PI-level fit
- Confident-sounding guesses about unverifiable facts
- Reddit hot takes treated as verdicts
- Questions without recommendations attached
- Bundling 5+ questions into one turn
- Asking for info already in the CV
- Writing SoP prose for the user (sop-coach scaffolds and critiques only)
- Cliché-padded essays ("passion for physics", "since childhood")
- AI-stilted cold emails sent without user revision
- Cold-emailing PIs at programs not yet evaluated
- Letting `profile.md`, `programs.md`, or `pi-contacts.md` go stale
- Scope creep in `grad-school-app` into specialist work
